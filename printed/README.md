# Printed Materials

Source files and print-ready PDFs for materials that ship with the OcuTrap R1 or
that customers print themselves. Keep all four artifacts in sync when any one of
them changes.

## Contents

| File | Purpose | Source of truth |
| --- | --- | --- |
| `R1_Manual_v2.docx` | Printed Installation & User Manual shipped with the trap | Edited in Word |
| `R1_Manual_v2.pdf` | PDF export of the printed manual (20 pages) | Generated from the `.docx` |
| `R1_Cheat_Sheet.pdf` | One-page letter-size quick reference (setup, buttons, LEDs) | Generated by `scripts/build_cheat_sheet.py` |
| `inside_sticker.png` | LED/button reference graphic that lives inside the POD | Design file source lives with the hardware team |

For the full online Knowledge Base PDF (50 MB, covers every KB article), see
`../OcuTrap_Knowledge_Base.pdf` built by `../scripts/build_pdf.py`.

## Rebuilding

From the repo root:

```bash
# Rebuild the cheat sheet PDF from the Python source
pip3 install --user reportlab Pillow
python3 scripts/build_cheat_sheet.py

# Rebuild the manual PDF from the .docx (requires pandoc + Chrome)
brew install pandoc   # one-time
./scripts/build_manual_pdf.sh
```

## When to update

- **Firmware changes that add/change LED patterns or button sequences**:
  update `R1_Manual_v2.docx`, re-export PDF, and update the action/state tables
  in `scripts/build_cheat_sheet.py`. Replace `inside_sticker.png` if the hardware
  team ships a new sticker design.
- **Hardware revision (new parts list, different assembly steps)**:
  update the `.docx`, re-export PDF. The cheat sheet usually doesn't need
  changes unless the Quick Start steps change.
- **Contact/URL changes** (support email, dashboard URL): update both the docx
  and the footer strings in `build_cheat_sheet.py`.
